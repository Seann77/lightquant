# LightQuant 后续阶段路线

更新时间：2026-06-13

本文记录后续阶段方向，用于排期和交接。具体实施仍以每个阶段开始前的提示词和当前代码状态为准。

## 当前基线

当前项目已经具备以下 MVP 能力：

- 登录/session、用户状态、协议同意记录。
- Supabase PostgreSQL 持久化。
- 积分账户、积分流水、积分预占、AI 成功扣费、失败释放。
- 充值套餐、订单、mock 支付、支付宝/微信 provider 骨架、notify 验签模拟和幂等保护。
- MiMo Pro OpenAI-compatible 真实模型调用。
- 策略生成、代码解析、平台转换三类 AI Skill。
- 文件上传、文本解析、安全扫描和 AI `inputFileId` 接入。
- 只读后台 MVP。
- 本地检查、低成本回归、真实 AI 验收和支付专项回归脚本。

## 阶段路线

### 第八阶段：真实支付接入准备

目标：

- 支付配置集中管理。
- 订单状态流转约束。
- 超时订单关闭。
- `payment_transactions` 审计字段。
- 后台订单排查字段。
- mock notify 幂等和金额校验。

当前状态：

- 已完成，并已通过 `npm run smoke:payment:regression` 覆盖。

### 第九阶段：真实支付接入 MVP

目标：

- 支付宝跳转支付 provider。
- 微信 Native 扫码支付 provider。
- 真实 notify route、验签、金额校验、幂等和积分到账。
- 前端 `paymentAction` 展示：支付宝新页面、微信二维码、轮询状态。

当前状态：

- provider、notify route、签名模拟、前端 action 和状态轮询已具备。
- 尚需使用真实支付宝/微信商户环境完成小额订单端到端验收。

验收重点：

- 创建真实支付订单。
- 用户完成支付后服务端 notify 验签成功。
- 订单变为 `PAID`。
- 充值积分到账。
- 重复 notify 不重复入账。
- `return_url` 不参与到账判断。

### 第十阶段：后台审计与手动补偿

目标：

- 后台审计日志。
- 只读排查升级为受控写操作。
- 手动补偿积分或修正异常订单的最小闭环。

边界：

- 必须先明确账务规则、审批规则和幂等规则。
- 所有改账必须 append-only，不能直接改旧流水。
- 后台写操作必须记录操作者、原因、来源单据和前后状态。

### 第十一阶段：任务队列/Worker

目标：

- 将 AI 长任务从 Web 请求生命周期拆出。
- 引入队列、Worker、任务重试和死信处理。
- 支持任务取消、超时释放预占、失败原因审计。

边界：

- 积分预占、确认扣费、失败释放仍复用现有 credit service。
- Worker 不应绕过 repository/service 账务规则。

### 第十二阶段：对象存储与文件生命周期

目标：

- 上传文件从数据库文本存储迁移到 Supabase Storage 或对象存储。
- 数据库保留文件元数据、hash、扫描结果和审计字段。
- 支持文件过期、删除策略和访问授权。

边界：

- 前端不能直传未授权对象。
- 文件进入 AI 前仍必须经过服务端扫描和输入限制。

### 第十三阶段：模型效果与 Skill 迭代

目标：

- 优化策略生成、代码解析、平台转换的成功率和输出质量。
- 强化 JSON 输出稳定性。
- 建立典型样例集和回归用例。

边界：

- 不把完整服务端 skill prompt 暴露给前端。
- 模型失败不扣费，成功返回才确认扣费。

### 第十四阶段：运营指标与成本看板

目标：

- 用户增长、充值、AI 调用、token 成本、失败率、文件风险等运营指标。
- 后台成本看板。
- 支持按天、渠道、任务类型统计。

边界：

- 看板只展示必要信息，不展示密钥、完整用户代码或敏感上传内容。

### 第十五阶段：正式部署与安全加固

目标：

- 正式服务器或容器部署。
- HTTPS、反向代理、环境变量、日志、备份和监控。
- 生产禁用 mock 数据、mock 支付、mock AI。
- 安全加固：限流、CSP、IP 白名单、告警、备份演练。

边界：

- 部署前必须通过生产配置检查。
- 不允许把 `.env.local`、数据库连接串、AI Key、支付私钥、短信 AccessKey 写入仓库。

## 最近推荐下一步

1. 先完成真实支付宝或微信小额订单端到端验收。
2. 再进入后台审计与手动补偿阶段，补齐客服查账和异常处理闭环。
3. 随后引入队列/Worker，降低 AI 长任务对 Web 进程的依赖。
