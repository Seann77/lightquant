# AGENTS.md

## 项目定位

LightQuant 是一个面向 C 端和 B 端用户的量化策略修改 SaaS 平台，当前处于内测阶段。项目重点是建设网站、应用能力、AI 工作台、任务链路、积分账本、文件上传、支付骨架和后台管理，而不是直接提供投资建议或代替用户做交易决策。

本项目支持或计划支持 PTrade、JoinQuant、QMT 等量化平台的策略生成、修改、迁移和解释。所有量化相关功能应定位为策略工程辅助、代码转换辅助和逻辑审阅辅助。

## 当前阶段

- 当前主要工作是继续建设和稳定 SaaS 网站及应用功能。
- 优先保证内测阶段的核心用户路径可用：登录、AI 任务、文件上传、积分、支付、后台只读管理、结果展示和历史恢复。
- 新功能应尽量小步交付，避免一次性改动多个业务域。
- 涉及真实用户数据、积分扣费、支付回调、数据库结构、AI 调用成本、上传文件安全和权限控制的修改，应先识别影响范围，再谨慎实现。

## 开发工作原则

- 默认使用中文沟通，除非用户明确要求其他语言。
- 修改前先阅读相关代码、文档和已有实现，优先沿用当前项目结构、命名、组件风格、服务边界和验证方式。
- 保持改动范围小而清晰，不做无关重构，不大面积格式化文件，不顺手改变用户未要求的行为。
- 不要覆盖、删除或回滚用户已有改动，除非用户明确要求。
- 前端开发应服务于真实 SaaS 工作流，界面保持专业、清晰、可操作，避免把应用页面做成营销落地页。
- 后端开发应优先保持幂等、审计、错误可追踪和失败可恢复。
- 修改完成后运行与改动范围匹配的最小必要验证；无法验证时说明原因和残余风险。

## 高风险业务边界

以下区域属于高风险改动，修改前必须先理解现有链路和测试脚本：

- `src/server/ai/`：AI provider、任务 runner、输出合同、续写、完整性校验、计价和技能逻辑。
- `src/server/credits/`：积分预占、扣费、释放、幂等和账本记录。
- `src/server/payments/`：支付配置、订单创建、notify 回调、签名校验、幂等保护和异常补偿。
- `src/server/files/`：上传规则、文件扫描、代码安全和文件访问边界。
- `src/server/auth/`：登录、会话、短信、当前用户识别和权限判断。
- `src/server/admin/`：后台权限、只读管理、运营配置和审计入口。
- `prisma/`：数据库 schema、迁移、seed 和生产兼容性。
- `.env.example`、部署脚本和 CI 检查：不要引入真实密钥或破坏部署路径。

除非用户明确要求，不要随意改变价格、套餐、积分扣费规则、支付状态机、用户权益、数据库迁移策略或平台支持范围。

## AI / Skill 修改规则

量化策略领域规则主要由应用内的 AI skill、prompt、domain rules 和相关服务维护。开发时不得绕过、削弱或删除这些规则。

当任务涉及 `src/server/ai/skills/`、AI prompt、策略生成、策略修改、代码转换、策略解释、输出格式、平台适配或领域规则时，必须遵守以下边界：

- 保留用户原始策略意图，除非用户明确要求优化收益、换策略、调参或改变交易逻辑。
- 不主动改变选股逻辑、买入条件、卖出条件、仓位规则、风控逻辑或调仓周期。
- 不编造平台 API、字段或生命周期函数；不确定时先查项目内已有用法、用户提供材料或官方资料。
- 严格警惕未来函数、偷看未来数据、回测与实盘语义混淆、股票代码格式不一致、交易日和时间处理错误、现金和持仓字段误读。
- 面向非程序员解释策略时，用自然语言说明策略目标、选股条件、买入规则、卖出规则、仓位控制、风控、数据依赖和潜在风险，避免堆砌变量名和 API 名。
- 所有量化相关输出只能作为代码、逻辑和工程分析，不构成投资建议，不承诺收益。

如果只是开发网站、页面、后台、支付、积分、上传或通用工程能力，不应把上述策略细则机械套用到所有代码，但也不能破坏这些规则的承载链路。

## 数据、安全与合规

- 不读取、不输出、不提交 `.env.local` 中的真实密钥、数据库连接串、模型 Key、短信 AccessKey、支付私钥或生产凭据。
- 不把真实用户数据、订单数据、上传文件内容或后台敏感信息写入公开文档、日志样例或测试夹具。
- 支付 notify、积分账本、订单状态和 AI 任务记录必须保持幂等和可审计。
- 上传文件相关逻辑必须考虑大小限制、类型限制、危险代码、隐私内容和错误提示。
- 产品文案不得承诺收益，不得暗示平台替用户做投资决策。

## UI / 产品体验原则

- 这是内测期 SaaS 应用，优先建设真实可用的工作台和管理流程，不做空泛展示页。
- 页面应清楚表达当前任务状态、积分消耗、文件上传状态、AI 运行结果、失败原因和下一步动作。
- 保持现有视觉系统和组件风格，优先复用 `src/components/ui/`、Workbench、Shell 和已有业务组件。
- 与支付、积分、AI 调用、文件上传有关的操作应给出清晰状态反馈，避免用户重复提交或误以为扣费失败。

## 验证命令

根据改动范围选择最小必要验证：

```bash
npm run check:quick
npm run typecheck
npm run db:validate
npm run check:secrets
npm run smoke:ui:pages
npm run smoke:ai:output-contracts
npm run smoke:ai:domain-rules
npm run smoke:payment:regression
npm run smoke:local:regression
npm run build
```

常见选择：

- 普通前端或轻量服务改动：优先 `npm run check:quick`，必要时加 `npm run typecheck`。
- Prisma 或数据库相关改动：运行 `npm run db:validate`，必要时补充对应 smoke。
- AI skill、prompt 或输出合同改动：运行 `npm run smoke:ai:output-contracts` 和 `npm run smoke:ai:domain-rules`。
- 支付、积分、订单状态改动：运行相关 payment smoke，影响较大时运行 `npm run smoke:payment:regression`。
- 发布、密钥、环境变量或公开仓库前检查：运行 `npm run check:secrets`。

如果验证依赖本地服务、数据库或外部配置而无法运行，必须在最终说明中写清楚。

## 不要做的事

- 不要把本项目改造成单纯的策略收益优化工具。
- 不要在没有用户要求时主动改变策略业务语义。
- 不要绕过积分、支付、权限、文件安全或 AI 输出完整性检查。
- 不要提交真实密钥、真实用户数据、真实订单数据或生产凭据。
- 不要手工编辑 `src/generated/prisma/` 等生成文件，除非明确是在处理生成产物异常并说明原因。
- 不要执行破坏性 git 操作，除非用户明确要求。
