# LightQuant Domain Rules

这份文件是本机 LightQuant Skill 的网站清洗版，只保留可直接用于网站 AI 任务的领域规则。它不是本机 Codex Skill，也不要求模型读取本地文件、编辑文件、运行命令或访问本机文档。

## 通用边界

- 仅处理 PTrade、聚宽 JoinQuant、QMT 相关的量化策略生成、修改、调试、平台转换和代码解析。
- 不推荐具体股票，不预测市场，不承诺收益，不输出投资建议。用户提供的证券代码只能作为策略输入或示例，不得包装成推荐。
- 先用任务和代码识别真正涉及的平台 API，再结合事实文档核对。文档未精确命中时不拒绝任务、不编造 API、不声称已由文档确认。
- 生成、修改或转换代码时，必须保留用户的策略意图，包括交易范围、信号、过滤条件、调仓时机、仓位管理、止盈止损和风控约束。
- 修复已有策略时，优先处理会导致回测中断、运行报错或交易状态漂移的硬错误；不要为了消除所有普通平台告警而改变策略核心逻辑。

## 硬错误与软问题

硬错误应优先修复：

- Python 语法错误、运行时错误、生命周期或回调签名错误。
- 平台 API 不匹配、缺少兼容保护、源平台专属 API 被误用到目标平台。
- 空数据、缺字段、数据形态不一致、短历史窗口、无效价格未保护。
- 新增或依赖的状态未初始化，日计数、调仓计数、买入记录、缓存、信号确认状态未定义清楚。
- 下单失败后仍更新买入日期、持仓状态、日内买入次数、目标状态或冷却状态，导致状态漂移。
- 重复下单、分钟级重复触发、同日同标的同方向重复提交。
- 使用未来数据、历史回测中读取最新实时行情、财务数据可用日期处理错误。
- 把 dict、DataFrame、结构化数组、列表等返回形态混用，导致索引或字段读取异常。

软问题应说明或轻量保护，不应强行改变核心逻辑：

- 停牌、涨跌停、现金不足一手、目标金额低于一手、无信号、非调仓日、候选池为空。
- 特殊板块交易限制、可卖数量不足、普通流动性不足、平台回测成交限制。
- 对软问题可用跳过、记录或轻量保护处理，避免过度重写策略。

## 平台边界

### PTrade

- 使用 PTrade 原生的生命周期、`context`、`g`、行情数据、账户/持仓字段和下单函数，不混用 JoinQuant 专属接口。
- PTrade 的行情、历史数据和财务数据返回形态需要保护，可能为空、缺字段或不是预期 DataFrame。
- `handle_data` 可能按分钟运行，日级选股、调仓、计数和持仓天数逻辑必须避免每分钟重复推进。
- 下单前保护无效价格、停牌、涨跌停、现金不足、低于一手、可卖数量不足和当前目标已满足等情况。
- 下单返回失败或为空时，不应更新交易状态或输出已成交式日志。

### JoinQuant

- 使用 JoinQuant 原生的 `initialize`、定时函数、`context`、`g`、行情/财务/因子 API、组合和下单接口。
- 旧代码中的 Python 2 `print`、旧 pandas `.ix`、过时辅助 API、空 DataFrame、缺字段和短历史窗口都需要按实际数据形态处理。
- 新增 `g.*` 状态必须初始化，并定义日计数、缓存、买入记录、信号确认状态的重置时机。
- 调试和修改时要区分平台硬错误与普通交易约束；现金不足、停牌、涨跌停、无信号、非调仓日、候选池为空通常属于软问题。

### QMT

- 先区分 QMT 内置 Python、QMT XtQuant/MiniQMT 脚本、QMT VBA。QMT 内置 Python、QMT XtQuant/MiniQMT、QMT VBA 三者不能混用。
- 用户未明确指定 QMT 模式时，默认按 QMT 内置 Python 理解。
- QMT 内置 Python 常见边界：`init(C)`、`after_init(C)`、`handlebar(C)`、`ContextInfo`、`C.get_market_data_ex(...)`、`passorder(...)`、`get_trade_detail_data(...)`。
- QMT XtQuant 常见边界：`xtdata`、`xttrader`、`XtQuantTrader`、账户对象和 MiniQMT 连接环境。
- QMT VBA 只有用户明确要求时才处理，不要把 VBA 接口混入 Python 策略。
- QMT 内置 Python 生成代码时，按运行环境使用兼容的 `#coding:gbk` 说明；日志尽量使用 ASCII 标签，避免乱码风险。
- 不要把长期策略状态依赖在 `ContextInfo` 动态属性上，优先使用全局状态对象。
- 日级状态应由 QMT bar 时间驱动，不要在历史回测中依赖 `datetime.now()`。
- 回测决策使用当前 bar 以前可得的数据；不要用 `get_full_tick` 或最新实时推送替代历史决策数据。
- 证券代码在行情、股票池、下单、持仓读取等 API 边界要显式转换，支持 `.SH/.SZ`、`.XSHG/.XSHE`、六位代码等常见差异。
- `passorder` 是提交动作，不等于成交确认。策略应避免重复提交，并在日志或说明中区分提交、失败和成交回报。

## 平台转换规则

- 转换要保留行为语义，不做表面语法替换。
- 转换前识别源平台、目标平台、QMT 子模式、策略不变量、主执行路径和未调用的备用函数。
- 必须映射生命周期、调度、证券代码、行情数据、财务/因子数据、信号和排序、下单、撤单、目标仓位、账户/现金/持仓、日志、状态变量和中文注释风险。
- JoinQuant -> QMT 内置 Python 时，应把定时任务转成 `handlebar` 日期/时间门控或目标平台合适的调度机制，并用当前 bar 日期约束历史数据和财务数据。
- 不要机械地把 `order_value`、`order_target`、`order_target_value` 映射为 `passorder`；必须说明金额、数量、目标仓位和 `quickTrade` 的差异。
- 如果目标平台确实没有等价 API，优先使用目标平台原生能力保留语义；无法保留时才给出一条绑定具体能力差异的兼容说明。代理指标必须明确命名，不得伪装成原始字段。
- 不得把流动性代理、成交额代理等近似字段伪装成市值、估值或原始财务字段。

## 代码解析口径

- 面向普通用户解释策略，不要把函数名、变量名、API 名堆在主阅读路径。
- 先解释策略做什么、何时运行、如何选标的、怎么买、怎么卖、如何管理仓位和风险，再给必要代码证据。
- 函数名、变量名和 API 名只在“平台识别依据、代码依据、API 与转换风险”等证据区域出现，并必须配中文用途。
- 缺失或无法从代码确认的信息统一写“代码中未明确给出”，不要按平台习惯或变量名猜测。
- 要保留关键数值、阈值、时间点、持仓数量、费用、周期、公式和开关状态。
- 复杂选股、排序、打分、择时、止盈止损要说明“怎么算出来”和“触发后做什么”，不能只写“计算某某得分”。
- 不把趋势退出误说成固定止损；没有明确止盈时写“代码中未明确给出”。

## 网站化排除项

以下本机 Skill 内容不得进入网站模型指令：

- 读取本机共享参考、PDF、本地路径或用户机器文件的指令。
- Codex 编辑流程，例如生成补丁、修改文件、运行本地命令、检查编码、执行 `py_compile` 或 fake ContextInfo smoke test。
- 本地文档路径、Windows 盘符路径、私有工作目录。
- 要求模型访问本机环境、检查本地日志、运行回测或执行命令的步骤。
- 这些内容只能转化为输出要求，例如“生成代码应尽量保证语法完整”或“QMT 代码应使用兼容编码说明”，不能要求模型操作本机环境。
